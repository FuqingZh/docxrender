# docxkit Minimal DOCX Rendering Core

Status: Draft

Date: 2026-06-25

Scope: internal Python package for DOCX/PDF technical rendering only.

## Summary

`docxkit` is a small technical package for Word-first report rendering. Its
first contract is intentionally narrow:

```text
template.docx + context + markdown_body + DocxStyle -> report.docx -> report.pdf
```

The package owns DOCX template rendering, markdown body insertion, Word style
application, DOCX field refresh/freeze handling, and LibreOffice-based PDF
conversion. Product repositories own report content, workflow layout,
manifest schemas, figure selection, captions, and delivery directory semantics.

The first implementation should prove the package boundary and API shape before
porting large amounts of code.

## Problem

`proteomics`, `trait_association`, and `joint-proteome_phospho` each carry
similar report rendering code:

- DOCX template rendering through `docxtpl`.
- Markdown body insertion through `python-docx`.
- Word style handling for headings, paragraphs, figures, notes, and tables.
- DOCX field refresh/freeze behavior for tables of contents and page fields.
- LibreOffice PDF conversion and failure diagnostics.

The duplicated code is not purely technical today. Product-specific report
content is mixed with reusable rendering mechanics. That makes fixes difficult
to share, and it encourages technical rendering helpers to learn product
concepts such as report sections, result directories, figure registries, or
workflow resource paths.

## Non-Goals

`docxkit` minimal core must not define or own:

- Report manifest schemas.
- Workflow resource layout contracts.
- Jinja section discovery.
- `Result/...` path semantics.
- `结果目录` text generation.
- Product-specific context builders.
- Product figure registries.
- Figure captions.
- Product section rules.
- Config-file reading in public API.

Callers may store style or report configuration in TOML, JSON, YAML, databases,
or code. `docxkit` receives structured Python objects and does not own that I/O.

## Public API Contract

The first public API exposes two operations.

```python
write_docx(options: DocxWriteOptions) -> DocxWriteResult
convert_docx_to_pdf(options: DocxToPdfOptions) -> DocxToPdfResult
```

### Public Dataclasses

```python
DocxStyle
DocxFontStyle
DocxSizeStyle
DocxTableStyle
DocxParagraphStyle
DocxFieldRefreshOptions
DocxWriteOptions
DocxWriteResult
DocxToPdfOptions
DocxToPdfResult
```

### Naming Rationale

- Use `write_docx` because the operation persists a DOCX file.
- Use `convert_docx_to_pdf` because the operation converts one document format
  to another.
- Avoid public `read_report_style`; callers own config file format and I/O.
- Avoid public `render_sections`; callers own workflow template layout and
  product content.

### Public Input Boundary

`DocxWriteOptions` should contain only technical rendering inputs:

- `file_template: Path`
- `file_out_docx: Path`
- `context: Mapping[str, Any]`
- `markdown_body: str`
- `dir_base: Path`
- `style: DocxStyle`
- `anchor_token: str`
- `should_update_fields: bool`
- `should_freeze_fields: bool`
- `field_refresh: DocxFieldRefreshOptions | None`

`DocxFieldRefreshOptions` should contain only technical LibreOffice/UNO refresh
inputs:

- `exe_libreoffice: Path`
- `dir_user_profile: Path`
- `file_out_docx_refreshed: Path | None`
- `file_listener_log: Path | None`
- `should_require_toc: bool`
- `should_freeze_fields: bool`
- `timeout_seconds: float`
- `poll_interval_seconds: float`
- `stable_checks: int`

`DocxToPdfOptions` should contain only conversion inputs:

- `exe_libreoffice: Path`
- `file_in_docx: Path`
- `file_out_pdf: Path`
- `dir_user_profile: Path`
- `file_out_docx_refreshed: Path | None`
- `file_listener_log: Path | None`

The public API must not accept product IDs, report roots, section file lists,
manifest paths, workflow stage roots, or result directory policies.

## Internal Layers

The supported public API is the package root export list in `docxkit.__all__`.
Implementation modules use semantic technical names instead of a blanket
leading-underscore convention:

- `docxkit.api`
- `docxkit.markdown`
- `docxkit.docx`
- `docxkit.pdf_uno`

These modules are implementation layers, not compatibility-stable public API.
Product repositories should import only from `docxkit` unless a later design
document explicitly promotes a submodule contract.

Internal state objects, when used, are implementation details and not public
API. The DOCX writer should prefer local variables over a state object while
the flow remains small.

## DOCX Pipeline

The DOCX writer should be implemented as a shallow, linear main path. This
keeps the flow readable and prevents hidden cross-module state.

```python
_write_template_docx(options)
markdown_blocks = parse_markdown_blocks(options.markdown_body)
document = Document(str(options.file_out_docx))
insert_markdown_blocks(document, markdown_blocks, ...)
document.save(str(options.file_out_docx))
if options.should_update_fields:
    write_docx_field_update_markers(options.file_out_docx)
if options.field_refresh is None and options.should_freeze_fields:
    write_frozen_docx_fields(options.file_out_docx)
refresh_docx_fields(options.file_out_docx, options=options.field_refresh)
result = DocxWriteResult(file_docx=options.file_out_docx)
```

Use module-level functions for semantic steps. Do not add wrapper steps that
only instantiate a state object or forward one field into a result object.

`refresh_docx_fields` must be a no-op when `DocxWriteOptions.field_refresh` is
`None`. When configured, it may use the optional LibreOffice UNO runtime to
write a refreshed DOCX, validate materialized TOC results, and freeze refreshed
field results.

## PDF Pipeline

The PDF converter should stay separate from the DOCX writer. It has a different
resource lifecycle because it owns a LibreOffice process, a UNO connection, and
temporary staged files.

PDF conversion and DOCX field refresh are optional UNO-backed features. Base
`docxkit` installation must support DOCX writing without importing UNO when
`field_refresh=None`. `docxkit` must not expose a `docxkit[pdf]` extra unless a
future implementation has a real Python-side runtime dependency to install.
LibreOffice and UNO bindings remain external runtime dependencies supplied by
the operating system, a container image, Conda environment, or LibreOffice
distribution.

Expected internal steps:

- `create_docx_to_pdf_state(options)`
- `validate_docx_input(state)`
- `copy_docx_to_stage(state)`
- `create_libreoffice_listener_command(state)`
- `run_libreoffice_listener(state)`
- `load_uno_document(state)`
- `refresh_uno_document_fields(state)`
- `convert_uno_document_to_pdf(state)`
- `copy_refreshed_docx(state)`
- `create_docx_to_pdf_result(state)`

The implementation should preserve staged input handling so LibreOffice does
not open the source DOCX in place.

## Style Defaults

Default style values are modeled after:

```text
/home/fqzhang/project/workflows/resources/common/report/style.toml
```

That file is a reference for the field model and initial default values:

- Fonts.
- Title, body, caption, table, and heading sizes.
- Table border color and border sizes.
- Table stripe fill color.
- Table line spacing.
- Body and note paragraph line spacing.
- Body paragraph first-line indent.

`docxkit` public API accepts `DocxStyle` directly. TOML/JSON/YAML/config
adapters belong in caller repositories or a later optional helper module, not
in the minimal core.

## Caller Responsibilities

Product repositories must prepare:

- Structured `DocxStyle`.
- `context` for `docxtpl`.
- Already-rendered `markdown_body`.
- Template DOCX path.
- Asset base directory.
- Output paths.

Product repositories also own:

- Jinja section rendering.
- Fragment fallback conventions.
- Product-specific content and wording.
- Manifest and result-package contracts.
- Figure discovery and captions.
- `Result/...` path policies.
- `结果目录` text, if a product wants that text in markdown.

## Test Plan

The first implementation should add tests before migrating product wrappers.

Required tests:

- `DocxStyle` dataclass construction from caller-provided values.
- Minimal `write_docx` using a fixture DOCX template and markdown body.
- Markdown headings, paragraphs, ordered lists, images, tables, and hard line
  breaks.
- Note paragraph style as configurable formatting, not business text
  generation.
- DOCX field update/freeze XML behavior.
- `convert_docx_to_pdf` command and UNO failure diagnostics with mocks.
- No product repository imports inside `docxkit`.

Product compatibility tests should come after the minimal package tests pass.

## Compatibility Boundary

The first release should be additive and conservative. Existing product
repositories should keep their current report modules until wrapper migrations
prove equivalent behavior.

No product repository should import implementation modules such as
`docxkit.markdown`, `docxkit.docx`, or `docxkit.pdf_uno`.
If a product needs a lower-level feature, first decide whether it belongs in
`DocxWriteOptions`, `DocxToPdfOptions`, or a later separate public contract.
