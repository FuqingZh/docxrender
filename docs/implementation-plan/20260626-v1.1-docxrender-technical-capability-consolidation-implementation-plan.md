# docxrender Technical Capability Consolidation Plan

Version: v1.1
Date: 2026-06-26
Status: current

Related architecture:
`../architecture/20260625-v1.0-docxrender-minimal-core.md`

## Summary

This plan defines the next `docxrender` consolidation line after the minimal
core. The goal is to move reusable DOCX/PDF rendering mechanics out of
`proteomics`, `trait_association`, and `joint-proteome_phospho` without moving
product report semantics into `docxrender`.

The preferred direction is:

```text
pure DOCX/PDF mechanics first
configurable technical policies second
markdown dialect integration last
product report content never
```

`DocxRenderer` remains the primary fluent pipeline entrypoint. Dataclass
options and module-level functions remain the explicit advanced API.

## Boundary Decisions

`docxrender` should own these technical concerns:

- DOCX style policies for paragraphs, runs, headings, captions, notes, and
  tables.
- DOCX field update markers, TOC/page-field refresh, and field freezing.
- LibreOffice/UNO DOCX refresh and PDF conversion diagnostics.
- Generic `docxtpl` template rendering from caller-provided context.
- Generic DOCX body anchor insertion when the caller explicitly provides the
  anchor token or selector.
- Common Markdown-to-DOCX block and inline rendering that does not depend on a
  product dialect.

`docxrender` must not own these product concerns:

- Report manifest schemas or workflow resource layout.
- Jinja section discovery and fragment fallback policy.
- `Result/...` path semantics or `结果目录` generation.
- Figure registry rules, figure selection, caption generation, or section
  ordering.
- Product context builders, report metadata routing, delivery package layout,
  or customer-facing text policy.

If a behavior depends on a product keyword, template placeholder, section name,
or report-specific paragraph meaning, the caller must pass that behavior as
configuration or pre-rendered content. It must not become a built-in default.

## Public API Direction

Keep the public surface small and pipeline-oriented:

- `DocxRenderer` is the recommended entrypoint for high-frequency users.
- `write_docx(DocxWriteOptions)` and `convert_docx_to_pdf(DocxToPdfOptions)`
  remain the explicit advanced API.
- Add public options only when they describe a reusable technical behavior, not
  a product convention.

Near-term public additions should prefer these shapes:

- `DocxRenderer.with_docx_style(...)` or existing style methods for formatting
  policies, if current style dataclasses are insufficient.
- `DocxRenderer.with_field_update_markers(...)` only if products need marker
  writing without LibreOffice refresh.
- `DocxRenderer.with_template_context(...)` only for generic `docxtpl` render
  inputs; product context construction stays outside.
- `DocxRenderer.with_body_anchor(anchor_token=...)` or an equivalent explicit
  anchor option for body insertion.

Avoid public methods whose name hides persistence or conversion. Persisted DOCX
output should stay under `write_docx(...)`; PDF output should stay under
`write_pdf(...)` on `DocxRenderer` and `convert_docx_to_pdf(...)` for the
advanced function API.

## Implementation Batches

### Batch 1: Pure Style And Field Capabilities

Intent: reduce duplication with the lowest semantic risk.

Changes:

- Expand `DocxStyle` only for technical formatting gaps proven by product code.
- Cover paragraph, run, table, caption, note, and heading formatting with tests.
- Add a small public field-marker configuration only if existing products need
  field markers without running LibreOffice.
- Keep all Word XML helpers internal.

Acceptance:

- Existing `docxrender` tests pass.
- New tests assert style and field behavior through public API only.
- Base DOCX writing still works without LibreOffice or UNO.

### Batch 2: Generic `docxtpl` And Body Anchor Handling

Intent: make template rendering and body insertion reusable without learning
product template names.

Changes:

- Keep `docxtpl` execution generic: callers provide template path and context.
- Add explicit body-anchor configuration if the current `anchor_token` contract
  is not enough for product templates.
- Preserve section properties when clearing or replacing a body anchor.
- Keep product context builders and section rendering in caller repositories.

Acceptance:

- Fixture DOCX tests cover anchor insertion and section-property preservation.
- Tests use neutral anchor tokens and do not reference product keywords.
- Product repositories import only from `docxrender` package root.

### Batch 3: Common Markdown-To-DOCX Parity

Intent: move only reusable Markdown rendering mechanics into `docxrender`.

Changes:

- Support common headings, paragraphs, hard line breaks, ordered lists,
  unordered lists, tables, images, and bold spans.
- Treat image width/alignment metadata as optional technical attributes only
  when passed in a generic syntax or pre-parsed structure.
- Keep product-specific paragraph classification, caption text generation, and
  result-directory rendering outside `docxrender`.

Acceptance:

- Markdown fixture tests cover each supported block and inline feature.
- No test depends on `结果目录`, `Result/...`, product section names, or product
  figure identifiers.
- Proteomics can replace its body insertion mechanics while keeping markdown
  assembly, figure registry, captions, and result-directory text locally.

### Batch 4: Product Compatibility Migrations

Intent: prove the boundary in real repositories after core parity exists.

Order:

1. Keep the current proteomics PDF/header-footer migration as the baseline.
2. Migrate proteomics body insertion only after Batch 3 parity tests pass.
3. Apply the same low-risk PDF/field/header-footer path to `trait_association`.
4. Audit `joint-proteome_phospho` for active source-level report rendering
   before treating it as a migration target.

Acceptance:

- Product tests continue to validate report order and generated DOCX assets.
- Product repositories do not import `docxrender` implementation modules.
- Any missing capability is documented as a public API gap before coding.

## Test Plan

Required `docxrender` tests:

- Public import and API shape tests for any added options or renderer methods.
- Style policy tests for paragraph, run, table, caption, note, and heading
  formatting.
- Field marker, refresh, freeze, and missing-runtime diagnostics tests.
- Template fixture tests for anchor insertion and section-property preservation.
- Markdown fixture tests for common blocks and inline spans.
- Negative tests proving product repositories are not imported by `docxrender`.

Required product validation:

- Proteomics report-render smoke tests after each migration batch.
- Trait-association report-render smoke tests before and after its migration.
- Source search checks to ensure product code imports only public `docxrender`
  symbols.

Standard verification commands should remain:

```bash
pdm run ruff check .
pdm run pyright
pdm run ty check
pdm run pytest
```

Product repositories may use their own `PYTHONPATH=src pdm run ...` commands
when validating migrations.

## Assumptions

- `docxrender` is still early enough to refine `DocxRenderer` without keeping
  compatibility aliases for removed fluent methods.
- Product repositories can keep their current markdown assembly and content
  builders while gradually replacing technical DOCX operations.
- Markdown dialect behavior should enter only through explicit configuration or
  caller preprocessing, not hardcoded product defaults.
- LibreOffice and UNO remain external runtime dependencies; `docxrender` only
  reports actionable diagnostics when those paths are used.
