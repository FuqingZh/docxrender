## Summary

This roadmap splits the first `docxkit` implementation into reviewable PRs.
The goal is to establish the correct technical boundary before moving large
amounts of report-rendering code out of product repositories.

The first implementation line must keep `docxkit` focused on:

```text
file_template + context + markdown_body + DocxStyle -> DOCX -> PDF
```

It must not introduce product report contracts, workflow resource layout rules,
section discovery, `Result/...` semantics, or style config file readers.

## PR1: Public Contract Skeleton

Intent: add the public API surface and type contract without implementing
rendering behavior.

Changes:

- Add public dataclasses:
  - `DocxStyle`
  - `DocxFontStyle`
  - `DocxSizeStyle`
  - `DocxTableStyle`
  - `DocxParagraphStyle`
  - `DocxFieldRefreshOptions`
  - `DocxWriteOptions`
  - `DocxWriteResult`
  - `DocxToPdfOptions`
  - `DocxToPdfResult`
- Add public function stubs:
  - `write_docx(options: DocxWriteOptions) -> DocxWriteResult`
  - `convert_docx_to_pdf(options: DocxToPdfOptions) -> DocxToPdfResult`
- Export only the public dataclasses and functions from `docxkit.__init__`.
- Add implementation module placeholders with semantic technical names.

Non-goals:

- No DOCX writing implementation.
- No PDF conversion implementation.
- No TOML/JSON/YAML style reader.
- No product compatibility wrappers.

Acceptance:

- Public imports work from `docxkit`.
- Type construction tests pass.
- No product repository modules are imported by `docxkit`.

Progress: complete.

## PR2: Minimal DOCX Write Pipeline

Intent: implement the smallest working `write_docx` path.

Changes:

- Add a shallow function pipeline:
  - `parse_markdown_blocks`
  - `insert_markdown_blocks`
  - `apply_docx_style`
  - `write_docx_field_update_markers`
  - `write_frozen_docx_fields`
- Support a fixture template DOCX with an anchor token.
- Support minimal markdown blocks:
  - headings
  - paragraphs
  - ordered lists
  - tables
  - images
  - hard line breaks

Non-goals:

- No field freezing yet.
- No LibreOffice/PDF conversion.
- No product-specific section rendering.

Acceptance:

- `write_docx` creates a readable DOCX from a fixture template.
- Markdown body content is inserted at the anchor.
- The main pipeline remains linear and shallow.

Progress: complete for the minimal write path.

## PR4: DOCX Field Update And Freeze

Intent: preserve table-of-contents and page-field behavior across Word/PDF
delivery.

Changes:

- Implement DOCX field update preparation.
- Implement optional field freezing controlled by `DocxWriteOptions`.
- Implement optional UNO field refresh controlled by `DocxFieldRefreshOptions`.
- Keep field XML helpers internal.

Non-goals:

- No PDF export.
- No public low-level field helper functions.

Acceptance:

- DOCX settings include field update markers when requested.
- Field freeze removes runtime field markup while preserving field results.
- `write_docx(..., field_refresh=None)` does not import UNO.
- `write_docx(..., field_refresh=...)` can write refreshed DOCX in place or to
  a separate output.
- Tests inspect DOCX XML directly.

Progress: complete.

## PR5: DOCX To PDF Conversion

Intent: implement `convert_docx_to_pdf` through LibreOffice UNO.

Changes:

- Add internal `DocxToPdfState`.
- Validate input DOCX.
- Stage source DOCX before opening it with LibreOffice.
- Start a LibreOffice listener.
- Load the staged document.
- Refresh fields.
- Convert to PDF.
- Optionally copy refreshed DOCX.
- Emit diagnostic-rich failures.

Non-goals:

- No command-line interface.
- No product wrapper integration.
- No conversion strategy beyond LibreOffice UNO.

Acceptance:

- Unit tests cover listener command construction.
- Unit tests cover load failure diagnostics with mocks.
- The implementation does not open the source DOCX in place.

Progress: complete. PDF conversion is exposed through the public
`convert_docx_to_pdf` API and guarded as an optional UNO runtime feature.

## Cross-PR Rules

- Keep public API small.
- Keep product content outside `docxkit`.
- Use module-level functions for domain behavior.
- Use `write_` for persisted DOCX output.
- Use `convert_` for DOCX-to-PDF conversion.
- Use `parse_`, `insert_`, `apply_`, `prepare_`, and `create_` according to the
  global naming rules.
- Do not expose markdown AST, Word XML helpers, or UNO helpers as public API.
- Add tests with each behavior change.
